## Test Procedures and Implementation

### Overview

Testing of the Privacy Chat application was carried out across three structured dimensions: unit testing to verify individual module behaviour in isolation, integration testing to validate end-to-end workflows spanning multiple components, and security testing to confirm that access controls, input validation, and security headers function correctly under adversarial conditions. A dedicated fourth dimension was reserved for ephemerality verification — the core privacy guarantee of the system — which was tested separately under every possible session termination scenario.

### Test Environment and Tools

All testing was performed on the development workstation running Windows 11 with WSL 2 (Ubuntu 22.04). The backend server was run locally on port 5000 using Node.js 18 LTS, connected to the MongoDB Atlas cloud cluster. The frontend Vite development server ran on port 5173 and proxied API requests to the local backend. API endpoint testing was performed using Postman, which was used to send crafted HTTP requests with specific headers, bodies, and authentication tokens to each route. Real-time Socket.IO behaviour was observed through the Network tab of Google Chrome DevTools, which displays WebSocket frames and event emission in real time. MongoDB collections were inspected during and after tests using MongoDB Compass, which was used to verify that private messages did not appear in the database under any tested termination scenario. No automated test framework such as Jest or Mocha was used; all tests were conducted as manual black-box tests against the running application.

### Unit Test Implementation

Unit testing focused on verifying the behaviour of individual API route handlers against valid inputs, boundary conditions, and invalid inputs. For the authentication module, each registration flow step was tested individually: the OTP dispatch endpoint was called with valid and duplicate email addresses to verify duplicate detection before an OTP is sent; the registration endpoint was called with valid OTP codes, expired codes, and incorrect codes to confirm that each produces the correct status code and error message; and the login endpoint was tested with correct credentials, wrong passwords, and non-existent email addresses. Password hashing was verified indirectly by confirming that the bcrypt comparison in the login handler correctly distinguishes valid and invalid passwords without exposing the hash value.

For the messaging module, text messages were tested at, below, and above the five-hundred-character limit to confirm that the limit is enforced at the route level. File upload tests were performed using Postman with multipart form data carrying image, document, audio, and video files of allowed and disallowed MIME types, and with files below and above the one-hundred-megabyte size limit. The message deletion endpoint was tested both by the original sender and by a different authenticated user to confirm that the four-hundred-three Forbidden response is correctly returned when a non-sender attempts to delete a message for everyone. The deterministic conversation ID generation function was verified by calling the utility with the two user IDs in both orders and confirming that the returned string was identical in both cases.

The private session module received the most rigorous unit-level attention. The session creation endpoint was called twice for the same conversation to confirm that duplicate creation is rejected with a four-hundred status. After sending a private message via the API, MongoDB Compass was used to query the messages collection by the message text content, confirming that zero documents were returned and that the message existed only in the server-side Map. The session end endpoint was called and the server-side Map was inspected via a debug log to confirm that the session entry had been deleted. File cleanup was verified by creating a temporary test file in the uploads directory, starting a private session, registering the file path with the session, ending the session, and then confirming that the file no longer existed on disk.

### Integration Test Implementation

Integration tests examined full user workflows from the initial HTTP request through Socket.IO event delivery to the final database or RAM state. The complete authentication flow was tested by walking through the entire registration process in a real browser session: dispatching the OTP, receiving the code via email, submitting the registration form, confirming the cookie was set in the browser's DevTools Application tab rather than in JavaScript-accessible storage, and then verifying that a subsequent page refresh maintained the authenticated session by observing the successful response from the `/api/auth/me` endpoint.

Real-time messaging was tested by opening two browser sessions simultaneously, one logged in as User A and one as User B, both connected to the same Socket.IO server. A message was sent from User A's session and the arrival time in User B's chat window was observed to confirm delivery latency was well under one hundred milliseconds in a local environment. The three-state read receipt progression was tested by observing that the sent checkmark appeared immediately on User A's message, the delivered state appeared as soon as User B's socket received the message event, and the read state appeared when User B opened the conversation.

The private session full lifecycle integration test was the most critical test in the suite. Both participants opened a shared private session in separate browser tabs. Five test messages with distinctive content phrases were sent. MongoDB Compass was used during the active session to confirm that none of the test phrases appeared in any document in the messages collection. The session was then ended by User A using the end-session button, and the server logs were observed to confirm that the Map entry was deleted. Compass was queried again after session end to confirm that the messages collection still contained zero matching documents.

The page-unload cleanup path was tested by starting a private session and then abruptly closing the browser tab rather than ending the session through the UI. The Beacon API sends a synchronous HTTP request to the cleanup endpoint during the unload event. The partner session in User B's window received the session-ended notification via Socket.IO within seconds of the tab close, confirming that the Beacon API path functions correctly even when the JavaScript context is being torn down.

### Security Test Implementation

Security testing was performed by crafting specific adversarial requests through Postman and the browser console. Access control on protected routes was tested by removing the authentication cookie from the browser's DevTools Application panel and then attempting to load the chat page, confirming an immediate redirect to the login page. The rate limiting behaviour was tested by writing a Postman collection runner that sent twenty-one sequential login requests within a fifteen-minute window; the twenty-first request received a four-hundred-twenty-nine Too Many Requests response with the expected error message, confirming that the express-rate-limit middleware was functioning correctly.

File upload security was tested by creating a test file with a `.exe` extension and a content type of `application/x-msdownload` and submitting it through the file upload endpoint. The Multer file filter rejected the upload with a four-hundred status response. Filename sanitisation was tested by uploading a file with a path traversal sequence in its original name; the sanitised filename stored on disk contained only the permitted characters, with all path separators replaced. The CORS enforcement was tested by sending a request to the API from an origin other than the configured frontend URL using the `Origin` header in Postman, which received a blocked response from the CORS middleware. Socket.IO authentication was verified by attempting a direct WebSocket connection without including an auth token in the handshake, which was rejected with an authentication error before the connection event was emitted.

### Ephemerality Verification Procedure

The ephemerality guarantee was verified under five termination scenarios. In each scenario, private messages with known distinctive content were sent, and MongoDB Compass executed a collection scan of the messages collection after the termination event. In every scenario — explicit session end by the user, browser tab closure via the Beacon API, forced WebSocket disconnect, Node.js server process restart, and multiple concurrent sessions ending simultaneously — the collection scan returned zero matching documents. The server restart scenario specifically verified the volatile nature of the JavaScript Map: after the Node.js process was restarted, the Map was re-initialised as empty, and only the session status records in the sessions collection remained, with no recoverable message content. This confirms that the architectural guarantee of ephemerality holds across all exit paths that were tested.
