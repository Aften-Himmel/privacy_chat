## 5.3) Data Dictionary

This section provides a complete field-level reference for every attribute stored across all MongoDB collections and the volatile in-memory private message store used by the Privacy Chat system.

---

### 5.3.1) Users Collection

| Field | Data Type | Constraints | Description |
|---|---|---|---|
| _id | ObjectId | Primary Key, auto-generated | MongoDB unique document identifier |
| username | String | Unique, required, trimmed, 3–20 chars | User display name used across the application |
| email | String | Unique, required, lowercase, trimmed | User email address used for login and OTP delivery |
| password | String | Required, min 6 chars | bcrypt-hashed credential (12 salt rounds); plaintext never stored |
| avatar | String | Optional, default empty string | Full URL path to the uploaded avatar image in /uploads |
| isOnline | Boolean | Default false | Set true when the user has an active Socket.IO connection |
| contacts | Array of ObjectId | References users._id | Array of mutually accepted contact user IDs; updated with $addToSet |
| publicKey | String | Optional | ECDH public key in JWK format string used for E2E key exchange |
| createdAt | Date | Auto-generated by Mongoose timestamps | Account creation timestamp |
| updatedAt | Date | Auto-generated by Mongoose timestamps | Timestamp of last profile modification |

**Indexes:** `{ username: 1, unique: true }`, `{ email: 1, unique: true }`

**Hooks:** `post('findOneAndDelete')` and `post('deleteOne')` cascade contact removal and message/file cleanup.

---

### 5.3.2) Messages Collection

| Field | Data Type | Constraints | Description |
|---|---|---|---|
| _id | ObjectId | Primary Key, auto-generated | Unique message identifier |
| conversationId | String | Indexed | For DMs: deterministic string formed by sorting and joining two user IDs with an underscore |
| groupId | ObjectId | References groups._id, optional | Populated for group messages; null for direct messages |
| sender | ObjectId | References users._id, required | The user who authored and sent the message |
| text | String | Default empty string, max 500 chars | Message body text content |
| fileUrl | String | Optional | Server URL path to the uploaded file attachment |
| fileName | String | Optional | Sanitised original filename (alphanumeric, period, underscore, hyphen only) |
| fileType | String | Optional | MIME type of the uploaded file (e.g. image/jpeg, application/pdf) |
| fileSize | Number | Optional | File size in bytes |
| deletedFor | Array of ObjectId | References users._id | IDs of users for whom this message has been soft-deleted; filtered out in queries |
| status | String | Enum: sent, delivered, read; default sent | Read receipt state progressed by Socket.IO events |
| readBy | Array of ObjectId | References users._id | IDs of users who have opened and read the message |
| createdAt | Date | Auto-generated | Message creation timestamp |
| updatedAt | Date | Auto-generated | Timestamp of last modification (e.g. status update) |

**Indexes:** `{ conversationId: 1 }`, `{ groupId: 1 }`

**Design Note:** Private (ephemeral) messages are never stored in this collection. They reside exclusively in the server-side RAM store defined in `socket/privateStore.js`.

---

### 5.3.3) Groups Collection

| Field | Data Type | Constraints | Description |
|---|---|---|---|
| _id | ObjectId | Primary Key, auto-generated | Unique group identifier |
| name | String | Required, trimmed, 1–60 chars | Group display name shown in sidebar and chat header |
| description | String | Optional, max 200 chars, default empty | Short description of the group's purpose |
| avatar | String | Optional, default empty | Full URL path to the group avatar image in /uploads |
| creator | ObjectId | References users._id, required | The user who created the group; only they can delete it |
| members | Array of ObjectId | References users._id | All current group members including the creator and admins |
| admins | Array of ObjectId | References users._id | Subset of members with elevated privileges (add/remove members, edit info) |
| createdAt | Date | Auto-generated | Group creation timestamp |
| updatedAt | Date | Auto-generated | Timestamp of last modification |

**Indexes:** `{ members: 1 }`

### 5.3.4) Invitations Collection

| Field | Data Type | Constraints | Description |
|---|---|---|---|
| _id | ObjectId | Primary Key, auto-generated | Unique invitation identifier |
| from | ObjectId | References users._id, required | The user who sent the contact invitation |
| to | ObjectId | References users._id, required | The user who received the contact invitation |
| status | String | Enum: pending, accepted, declined; default pending | Current lifecycle state of the invitation |
| type | String | Enum: normal, private, contact_request; default normal | Categorises the purpose of the invitation |
| createdAt | Date | Auto-generated | Invitation creation timestamp |
| updatedAt | Date | Auto-generated | Timestamp of last status change |

**Business Logic:** When a user sends an invitation, the backend checks for a reverse pending invitation from the recipient. If found, both are accepted automatically and mutual contacts are established without user interaction.

### 5.3.5) Sessions Collection (DM Private Sessions)

| Field | Data Type | Constraints | Description |
|---|---|---|---|
| _id | ObjectId | Primary Key, auto-generated | Unique private session identifier |
| conversationId | String | Required | Links the session to a specific DM conversation using the same deterministic ID format as the Messages collection |
| participants | Array of ObjectId | References users._id, exactly 2 elements | The two users participating in this private session |
| status | String | Enum: active, ended; default active | Session lifecycle state; only one active session per conversation is permitted |
| startedBy | ObjectId | References users._id | The user who initiated the private session |
| createdAt | Date | Auto-generated | Session start timestamp |
| updatedAt | Date | Auto-generated | Timestamp of last state change (typically when status becomes ended) |

**Indexes:** `{ conversationId: 1, status: 1 }`

### 5.3.6) GroupSessions Collection (Group Private Sessions)

| Field | Data Type | Constraints | Description |
|---|---|---|---|
| _id | ObjectId | Primary Key, auto-generated | Unique group private session identifier |
| groupId | ObjectId | References groups._id, required | The group to which this private session belongs |
| participants | Array of ObjectId | References users._id | Snapshot of Group.members copied at session creation time |
| status | String | Enum: active, ended; default active | Session lifecycle state; only one active session per group is permitted |
| startedBy | ObjectId | References users._id | The admin who initiated the group private session |
| createdAt | Date | Auto-generated | Session start timestamp |
| updatedAt | Date | Auto-generated | Timestamp of last state change |

**Indexes:** `{ groupId: 1, status: 1 }`

---

### 5.3.7) VerificationCodes Collection

| Field | Data Type | Constraints | Description |
|---|---|---|---|
| _id | ObjectId | Primary Key, auto-generated | Unique document identifier |
| email | String | Required, lowercase, indexed | The email address to which the OTP was sent |
| code | String | Required | Six-digit numeric one-time password |
| expiresAt | Date | Required, TTL index (expireAfterSeconds: 0) | The exact time at which MongoDB's TTL thread auto-deletes this document (10 minutes from creation) |
| createdAt | Date | Auto-generated | Document creation timestamp |
| updatedAt | Date | Auto-generated | Last update timestamp |

**Indexes:** `{ email: 1 }`, `{ expiresAt: 1, expireAfterSeconds: 0 }` (TTL)

**TTL Auto-Purge:** MongoDB's background TTL thread automatically deletes the document once `expiresAt` is reached, eliminating the need for a scheduled cleanup job.

### 5.3.8) Private Message In-Memory Store (RAM — Not a MongoDB Collection)

Private messages are deliberately excluded from MongoDB and reside only in a server-side JavaScript `Map` defined in `socket/privateStore.js`, keyed by session ID.

| Field | Data Type | Description |
|---|---|---|
| sessionId | String (Map key) | The session identifier that maps to an array of private message objects |
| _messageId | String | UUID generated via `crypto.randomUUID()` used to uniquely identify each message within the session |
| sender | Object | Sender identification object containing `_id` and `username` fields |
| text | String | Message text content |
| fileUrl | String (optional) | Temporary URL to the uploaded ephemeral file; deleted from disk when session ends |
| fileName | String (optional) | Sanitised original filename |
| fileType | String (optional) | MIME type of the uploaded file |
| fileSize | Number (optional) | File size in bytes |
| isPrivate | Boolean | Always true; signals the frontend to apply ephemeral UI styling |
| createdAt | String | Message creation timestamp in ISO 8601 format |

**Lifecycle:** The store is populated when a private session starts, messages are appended during the session, and the entire entry is destroyed via `Map.delete(sessionId)` on session end, socket disconnect, page unload (Beacon API), or server restart. Associated files on disk are concurrently removed via `fs.unlink()`.

### 5.3.9) Data Model Design Decisions

1. **Deterministic Conversation ID:** DM conversations are identified by a computed string formed by sorting the two participant user IDs alphabetically and joining them with an underscore (e.g. `abc123_def456`). This avoids a separate Conversations collection and makes the relationship implicit and queryable without joins.
2. **Polymorphic Message Collection:** A single Messages collection stores both DM messages (identified by conversationId) and group messages (identified by groupId), simplifying schema management and stream queries.
3. **Soft Delete via Array:** The `deletedFor` array enables per-user message visibility control without duplicating data or disrupting other participants' message history.
4. **Embedded Arrays over Junction Tables:** MongoDB's document model favours embedding related ObjectId references directly in documents (contacts, members, admins, readBy) rather than maintaining separate junction collections, reducing the number of collections and database round-trips.
5. **RAM-Only Private Store:** Using a server-side JavaScript Map for private messages provides the strongest possible guarantee of ephemerality — the data never touches any storage driver, disk buffer, or database write-ahead log.
